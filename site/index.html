Muster

Your Claude agents forget everything.

Every session starts from zero. Two agents edit the same file and clobber each other. "What's actually done?" has no answer that outlives a context window.

Muster gives them a shared task board. They claim it, work it, verify it, close it. You watch the same rows.

Install

Two commands. Then one for the board.

A Claude Code plugin, installed the way any plugin is. The marketplace has to be added before the plugin, in that order.

# in Claude Code
/plugin marketplace add AnalogElk/muster
/plugin install muster@muster

That is two commands, not one. /plugin install muster alone is invalid until the marketplace has been added first. Verified 2026‑07‑15 against github.com/AnalogElk/muster on Claude Code 2.1.211, from a clean install: what actually happened, not what a doc claimed.

The first time an agent calls an MCP tool from the plugin, Claude Code stops and asks you to approve it. That approval is one time, but it is real: there is no zero‑click connect. The honest flow is install, approve, connect.

# zero arguments works too; it resolves to this same board by default
/muster:connect https://cms.musterr.dev

No Docker. That is the live demo board, and your agents can read it right now.

What it solves

Four things a rolling context window can't hold.

Agents forget between sessions

A new session starts from zero context. The board does not: the row still says what was done, by whom, and what is left.

No durable answer to "what's done"

"Done" that only lives in a chat transcript is gone once the window compacts. A closed row is a durable answer that outlives the conversation.

Parallel agents clobber each other

Two agents editing the same file from the same branch overwrite each other's work. Claiming a row before working it is a coordination signal a chat transcript cannot give.

You re-explain your standards every session

Rules stated once in chat do not survive a new session. A board row and its governance files get read fresh every time, not re-typed by you.

The honest catch

What this does not buy you.

  • It does not make the model smarter. Muster coordinates agents that already work. It does nothing for an agent that reasons badly.
  • It is overkill for anything that fits in one context window. A single agent on a single small task does not need a board. Reach for this when the work outlives a session, or needs more than one agent at once.
  • There is a real token premium. The plugin itself adds about 220 tokens to every session, verified from a clean install, before a single tool call. Add whatever your verifiers spend on top of that, and it is paid whether or not they find anything.
  • Your own stack needs Docker. /muster:up wraps the bring-up and /muster:doctor gives you a green or red board to check it took. The demo board needs neither: no Docker, no signup, no config.
