# Claude-side OS — wired for {{BRAND_NAME}} ({{PROFILE}})

Generated by `elk-os wire` on {{GENERATED_AT}}. This directory is the
**Claude-side operating system** for your Elk OS deployment. Point Claude Code at
it and a session boots already wired to your deployment's shared-state board.

## What's here

| File | Purpose |
|---|---|
| `CLAUDE.md` | The constitution — governance every session follows (auto-loaded). |
| `.mcp.json` | The `elk-os-cms` MCP server, pointed at this deployment's Directus MCP endpoint. Reads the token from `${DIRECTUS_ADMIN_TOKEN}` in the environment. |
| `.claude/settings.json` | Registers the SessionStart hooks. |
| `.claude/hooks/session-banner.sh` | Prints the deployment banner + Directus/MCP health at session start. |
| `.claude/hooks/cms-task-snapshot.sh` | Prints the open `os_tasks` queue at session start — the agent side of the human <-> agent loop. |
| `.claude/memory/` | Memory seed (`MEMORY.md` index + the auto-memory pattern note). |
| `run-claude.sh` | Convenience launcher: loads the deployment `.env` into the environment, then `exec claude`. |

## Use it

The MCP server authenticates with `${DIRECTUS_ADMIN_TOKEN}`, which Claude Code
reads from its own environment. The simplest turnkey path:

```bash
cd {{WIRE_DIR}}
./run-claude.sh
```

`run-claude.sh` sources the deployment `.env` (which holds the real token,
gitignored) and launches Claude Code so the MCP server authenticates with no
manual export. The session-start hooks read the same `.env`.

To launch Claude Code yourself instead, make sure `DIRECTUS_ADMIN_TOKEN` (and
optionally `DIRECTUS_URL`) are exported, then run `claude` from this directory.

## The MCP endpoint

This deployment runs Directus's **native MCP server** at
`{{DIRECTUS_MCP_URL}}`. `elk-os wire` enables it (the `mcp_enabled` project
setting) so the `items` and `system-prompt` tools are live against your `os_*`
board. Re-run `elk-os wire` any time to regenerate this directory and re-assert
the setting.

> Secrets are never written into this directory. The token stays in the
> deployment `.env`; `.mcp.json` and the hooks only reference it by name.
